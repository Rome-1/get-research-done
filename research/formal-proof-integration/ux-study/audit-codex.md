# Codex Audit: `grd lean` CLI Surface

As a Codex runtime reading this, the first thing that struck me was that the implementation is more machine-structured than user-guided: most core paths return typed results with useful fields, but the CLI still exposes those results as generic tables or raw JSON and usually exits with a single failure code. That is good for agents that already know the schema, but a weaker first-run experience for an alternate runtime or human who needs the tool to say "this is a Lean diagnostic", "this is an orchestration failure", "bootstrap is still running", or "the faithfulness gate rejected this because the back-translation changed the quantifier".

## Cross-Perspective Notes

- As Codex, I look first for stable CLI contracts: flags, exit codes, JSON fields, and non-interactive failure recovery. A Claude Code reader may be more comfortable relying on rich prose in the agent definition and final dataclass dumps; I need the shell/API boundary to be explicit.
- The implementation reads as agent-internal plumbing with a helpful raw schema, but the command examples in `grd-prover.md` use `--json` while the actual CLI uses root `--raw`. That is exactly the kind of alternate-runtime mismatch that causes a tool call to fail before the model learns the intended contract.
- The faithfulness gate is conceptually strong, and the back-translation trace is the right primitive. From a Codex perspective, the rejection still needs an operational summary field, because parsing a large nested JSON/tree table to decide the next tool call is brittle.
- The daemon fallback policy is pragmatic for users, but it hides backend state changes. An alternate runtime trying to optimize latency or debug a flaky proof wants to know whether it used Pantograph, daemon-subprocess fallback, or direct subprocess without inferring that from a late `backend` string.
- The convention bridge is the clearest Claude-vs-Codex split: the agent brief says "generate this preamble", but the code path only prompt-injects convention values. I would not assume the formal artifact and the informal convention lock are actually coupled until a generated preamble file or explicit JSON field proves it.
